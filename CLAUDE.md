# AGENTS.md

> 本文件用于约束与指导自动化助手/代码协作者在本仓库中的行为与输出风格。  
> 核心原则：**最小化修改、可追溯、可回滚、避免引入不必要风险**。

## 1. 交互语言与回复格式（重要）

- **必须使用中文**与我对话与解释。
- **每次回复结尾必须加上**：`Nya～`
- 命令、路径、配置示例一律用代码块输出，避免混在段落里。

## 2. Git 操作限制（重要）

- **不要主动执行** `git add`  `git commit` 和 `git push`。
- 如需提交：只允许**提示我**应该如何 commit/push，由我来执行。
- 不要进行会造成大量 diff 的操作（例如全仓库格式化/重排 import），除非我明确要求。
- 每次任务结束前检查新增的代码/编译产物/二进制文件，将不需要git索引的文件添加到gitignore中。

## 3. 工作方式：先读再改（重要）

- 开工前先快速阅读：项目结构、入口文件、README/文档、已有测试方式。
- 不确定时先提出假设与依据，再给出最小改动方案；不要凭感觉大改。

## 4. 最小化修改原则（重要）

- 仅修改我明确要求或与修复直接相关的部分。
- **不要“顺手重构”**、不要改动未提及模块的风格/结构/命名。
- 若必须触及未提及部分（例如修复会影响到它）：
  - 先解释原因；
  - 将改动范围压到最小；
  - 明确说明影响与风险。

## 5. 模块化与文件体积控制（重要）

- 目标：**功能实现拆分为多个模块**，保持每个模块职责单一、边界清晰。
- 约束：**单个源代码文件不超过 1000 行**（包括注释/空行/类型声明等）。
- 若新增功能可能导致文件超 1000 行：
  - 优先拆分：抽取子模块/工具函数/子服务；
  - 保持 API 边界稳定；
  - 避免“大而全”的单文件堆砌。
- 在不违背“最小化修改”的前提下：拆分应只影响必要范围，不做无关重构。

## 6. API.md 作为索引（重要）

- 仓库中必须维护一个 `API.md`，用于**快速描述各模块做什么**以及**关键 API/入口的参数与返回**，便于 LLM 与人快速查找。
- **新增/修改模块功能时，必须同步更新 `API.md`**（保持简洁，覆盖关键点即可）。
- **做新功能/改功能的优先查找顺序**：
  1) **先读取 `API.md`**，定位对应模块与入口  
  2) 再进入对应模块文件查看实现细节  
  3) 最后才考虑全仓库搜索（仅在 API.md 缺失/不准确时使用）
- 本文件应保持“索引/目录”风格：短、准、可快速定位。

## 7. 临时测试文件管理（重要）

- 所有临时 test、debug、一次性脚本、验证文件等：  
  **必须放到 `/test_temp` 文件夹中**（如不存在就创建）。
- 不要在源码目录散落 `tmp_*`、`test_*`、`debug_*` 等文件。
- `/test_temp` 中的临时脚本命名建议：`YYYYMMDD_用途说明.ext`（示例：`20260306_repro_login_bug.py`）。
- 临时文件应可清理：在说明中明确“使用完是否可删除”。
- **必须将 `/test_temp/` 加入 `.gitignore`**，避免误提交。

## 8. 测试与验证（重要）

- 在完成我提出的所有需求之后，必须运行最相关的测试/验证步骤。
- **如果测试不通过：继续定位问题并修复，再次运行测试**，重复此流程，直到测试通过且没有已知问题为止。
- 如果无法运行测试（环境缺失/权限/依赖问题），必须：
  - 写明原因；
  - 给出我可以执行的验证步骤（命令/预期输出/判定标准）。

## 9. 日志与调试输出约束

- 临时 `print/log` 必须在最终方案中移除。
- 若必须保留调试输出，必须加清晰注释说明：用途、触发条件、如何关闭。
- 所有的结束报告都应该保存到 `/Report/`文件夹中 并 git ignore。

## 10. 错误处理与稳定性

- 不要用空 `except`、`try: ... except: pass`、吞异常来“修好”，除非我明确要求并说明原因。
- 新增容错逻辑应最小化，并附带可验证的失败/恢复路径。

## 11. 依赖与配置变更

- 除非我明确要求：
  - 不要新增依赖；
  - 不要升级依赖大版本；
  - 不要随意改动 lockfile（如必须改动，要解释原因与影响）。
- 示例配置中不要写入真实密钥或 token，使用 `ENV_VAR` 占位。

## 12. 安全与隐私红线

- 不要建议或生成会泄露密钥/隐私数据的做法。
- 不要把 token、cookie、密钥写进代码、文档、示例、日志输出。

## 13. 多文件修改前的清单（推荐）

- 若预计需要修改多个文件，先列出：
  - 将修改的文件列表；
  - 每个文件修改的目的；
  - 预期影响范围；
  - 回滚策略（如何撤销）。

## 14. 复用优先：不要重复造轮子（重要）

- 实现新功能或修复时，**优先复用现有模块/工具函数/公共组件**，避免复制粘贴相似逻辑。
- 在写新代码前，先做最小范围查找：
  1) 先查 `API.md` 找到可能的现有入口/模块  
  2) 再在对应模块内查找可复用的函数/类  
- 只有当现有实现无法复用或复用成本明显更高时，才允许新增重复功能；并说明原因。
- 抽取可复用代码时要遵守“最小化修改”，不要为复用而大范围重构。
- 优先调用MacOS系统自带API，搜索API手册后再生成代码。

## 15. .gitignore 维护（重要）

- 任何**不需要 Git 索引/不应提交**的文件或目录（临时文件、缓存、构建产物、下载数据、日志、用户数据等不应该公开的内容），**每次结束前检查并加入 `.gitignore`**。
- 新增忽略规则时：尽量精确（避免误忽略），并写清楚注释说明用途。

## 16. 代码块分区标记（重要）

- 生成的代码块应该添加上分区（python示例）
"
#region (具体分区)

#endregion
"

- 分区名要与内容对应，例如：
- #region 数据清洗
- #region 单元测试：登录流程
- #region 临时修复脚本

## 17. 设计和更新Plan.md（重要）

- `Plan.md`是用于给LLM提供项目规划的工具，用户不会阅读。
- 根据第一个初始指令生成`Plan.md`
- 每次任务结束之后将已经完成部分划掉。
- 每次开始前，你都应该阅读`Plan.md`
- 把当前工作进度和新的需求写入`Plan.md`，并删除已经不需要保留的工作进度
- `Plan.md`只应该存储核心架构和框架信息

## 18. Docs文件夹使用

- 将所有生成的手册等markdown文本的放入 `/Docs/` 文件夹中。

## 19. 设计Remember.md（重要）

- `Remember.md`是用于给LLM提供跨对话记忆的工具，用户不会阅读。
- 如果对话中有明确指示，把这些信息增加到`Remember.md`中，不应该超过3行。
- 对话结束后把未来可能用到的信息存入`Remember.md`，正文信息不超过30行。
- 每次对话结束后清理不需要记住和使用的信息，`Remember.md`不应该超过200行。
- 不要把 `Plan.md` 和 `Agents.md` 中的的内容重复写入。
- 每次开始前，你都应该阅读`Remember.md`。

## API.md（模块索引与接口摘要）

> 目的：为 LLM 与人类提供“快速定位入口”的索引文档。  
> 使用规则：做新功能/改功能时 **优先查阅本文件**，再进入对应模块阅读实现；仅在本文件缺失/不准确时再全仓库搜索。

### 模块索引（示例结构，按你的项目实际填）

#### <模块名/目录名>

- **职责**：一句话说明这个模块做什么
- **入口/API**：
  - `<函数/类/HTTP路由/CLI命令>`  
    - 参数：`a: 类型`，`b: 类型`（可选/默认值）  
    - 返回：`...`  
    - 说明：边界条件/异常/副作用（尽量短）
- **依赖/调用关系（可选）**：它主要被谁调用，或它调用谁
- **文件位置**：`path/to/module`

### 变更约定

- 新增模块/新增入口/变更参数：必须同步更新本文件。
- 本文件保持“索引风格”：宁可少，但要准；不要写长篇教程。
